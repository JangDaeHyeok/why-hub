# 구현스펙 — 관리자 승인 워크플로우

## 1. 목적·배경

지금까지 모든 쓰기는 `KnowledgeService.save_document` → `store.save_document`로 **즉시 반영·색인**됐다.
팀 내부 운영에서는 **모든 쓰기를 승인 대기 큐에 넣고, 관리자가 승인해야만 실제 반영**되도록 한다.
승인 전에는 지식 store·FTS 인덱스에 아무것도 쓰지 않으므로 검색/조회/목록에 노출되지 않는다.

- 불변식 §2-1: 실제 반영은 여전히 `store.save_document` 단일 경로. 승인 큐는 그 **앞단 스테이징**일 뿐.
- 불변식 §2-3: 승인 큐는 지식 콘텐츠가 아닌 **임시 워크플로우 메타** → knowledge store(delta 이력)와 분리된 별도 store.
- §5: MCP·HTTP·UI는 모두 `KnowledgeService`만 호출. 승인/제출 로직 중복 없음.

## 2. 결정 (확정)

- **범위:** 모든 쓰기(수동 편집·AI 적용·ingest)를 승인 대기. 관리자 포함(자기 제출을 자기가 승인 가능).
- **신원:** config `[approval] admins = [...]` 목록 + 기존 `actor` 필드. 실제 로그인 없음(현 MVP 신뢰 기반).
- **토글:** config `[approval] enabled`. **코드 기본값은 false**(opt-in — 기존 동작·seed·단위 테스트 무영향).
  팀 배포는 `config.example.toml` 처럼 `enabled = true` + `admins` 로 켠다. false면 기존처럼 즉시 반영.
- **삭제:** 하드 삭제 없음. "삭제" = `status: deprecated`로 바꾼 문서를 저장(op=deprecate).

## 3. 제출(Submission) 스키마

`knowledge/.submissions/<sub_id>.json` (파일 기반, FTS 인덱스와 분리 → 재색인 영향 없음):

```json
{
  "id": "sub-20260714-ab12cd",
  "op": "create | edit | deprecate",
  "doc_id": "adr-0004",
  "raw_markdown": "---\n...\n",
  "intended_diff": null,
  "actor": "alice",
  "status": "pending | approved | rejected",
  "prelint": {"ok": false, "reasons": ["대안 섹션 비어 있음"]},
  "created": "2026-07-14T10:00:00",
  "reviewer": null,
  "reviewed_at": null,
  "note": null
}
```

- `sub_id`: `sub-<날짜>-<랜덤6>`. 문서 id와 충돌하지 않는 별도 네임스페이스.
- `doc_id`: 정규화된 markdown의 frontmatter id에서 도출(없으면 제출 거부).
- `prelint`: 제출 시점 참고용(`_prelint` 재사용). **권위 있는 lint는 승인 시점의 실제 save가 수행**.

## 4. 상태 전이

```
(제출) --> pending --approve(admin)--> approved  [store.save_document 반영]
                    --reject(admin)---> rejected  [반영 없음]
```

- `pending`에서만 approve/reject 가능. 이미 처리된 제출 재처리는 거부(멱등·오작동 방지).
- approve 시 실제 save가 **lint 실패**하면: 제출은 **pending 유지**, 지식 store 변경 없음, 사유 반환.
- 상태 전이는 제출 잠금(`doc_lock`의 sentinel id `__submissions__`)으로 직렬화.

## 5. store 모듈 — `hub/store/submissions.py`

I/O + 원자적 쓰기(temp+rename)만. 순수 큐 store(정규화/lint/인덱스 모름).

- `create(root, *, op, doc_id, raw_markdown, intended_diff, actor, prelint, now) -> dict`
- `read(root, sub_id) -> dict | None`
- `list(root, status=None) -> list[dict]` (created 역순 정렬 — 최신 먼저)
- `set_status(root, sub_id, *, status, reviewer, note, now) -> dict`
- 경로 헬퍼는 `paths.submissions_dir/submission_path` 추가.

## 6. service API

- `submit_change(raw_markdown, *, actor, op="create", doc_id=None, intended_diff=None) -> dict`
  정규화로 doc_id 확정 → `_prelint` → `submissions.create`. 지식 store·인덱스 미변경.
- `list_submissions(status=None) -> list[dict]`, `get_submission(sub_id) -> dict | None`
- `approve_submission(sub_id, *, approver) -> SaveResult`
  `config.is_admin(approver)` 아니면 `PermissionError`. pending 아니면 `ValueError`.
  `store.save_document(raw_markdown, actor=제출자, ...)` 반영 성공 시 status=approved.
- `reject_submission(sub_id, *, approver, note="") -> dict` — 관리자 검증 후 rejected.
- 게이트 라우팅: `enabled`면 `save_document`/`ingest_source`가 직접 반영 대신 `submit_change` 반환(dict, `SaveResult` 아님).

## 7. 인터페이스별 동작

- **HTTP**: `PUT /docs/{id}`·`POST /ingest` → `{submission_id, status:"pending", prelint}` 반환(202).
  신규 `GET /submissions?status=`, `GET /submissions/{id}`, `POST /submissions/{id}/approve`(body: approver),
  `POST /submissions/{id}/reject`(body: approver, note). `PermissionError`→403.
- **MCP**: `save_document`/`ingest_source` → 제출 dict 반환. 신규 `list_submissions`/`approve_submission`/`reject_submission`.
- **UI**: `/ui/save` → 제출 후 "승인 대기" 안내. `/ui/approvals` 대기함(승인/반려 버튼, actor=검토자).

## 8. 테스트

- 제출 → pending, 검색/목록/조회에 안 뜸.
- 비관리자 approve → PermissionError/403. 관리자 approve → 반영·색인·검색됨.
- reject → 영구 미반영. deprecate 제출·승인 → status=deprecated, 이력 type=deprecation.
- 승인 시 lint 실패 → 제출 pending 유지, store 무변경.
- `enabled=False` → 기존 즉시 반영 경로(하위호환).
