# 구현 스펙 (초기 v0) — `/generate` : AI 문서 초안 생성 (M8)

> **상태:** 초기 초안(v0). 이 기능은 **프롬프트 반복 튜닝 대상**이라 문구는 확정이 아니라 시작점이다.
계약(입력/출력)·가드레일은 지키되, 프롬프트 본문은 골든 셋으로 개선해 나간다.
**범위:** 기획안 1 §6.2(경로 B) · §9.2(UI AI 생성) · §11(`POST /generate`)의 구현. M8.
**불변식:** 생성물은 **초안일 뿐**이며, 저장은 반드시 **사람 검토 + lint 게이트 + save 루틴**을 거친다.
> 

---

## 1. 목적 · 위치

UI에서 사용자가 소스를 주면, 서버가 LLM을 호출해 **필수 섹션이 채워진 ADR/설계의도/가이드 초안**을 만든다.
`/generate`는 **초안을 반환만** 한다(저장하지 않음). 사용자가 UI 에디터에서 검토·수정한 뒤 `save_document`로 저장.

## 2. 엔드포인트 계약

```
POST /generate
```

**입력**

```json
{
  "target_type": "adr",                 // adr | design-intent | guide
  "sources": [
    {"kind": "doc",    "id": "adr-0003"},         // 허브 내 기존 문서
    {"kind": "upload", "text": "<붙여넣은/업로드 원문>"},
    {"kind": "note",   "text": "<대화 요약 등>"}
  ],
  "hint": "인증 방식 결정을 ADR로"        // (선택) 사용자 지시/제목 힌트
}
```

**동작**

1. `sources`를 로드/취합한다(허브 문서는 store에서, upload/note는 그대로).
2. **관련 기존 ADR 자동 수집** — 리트리버(유사 RAG)로 `hint`/소스 키워드에 관련된 accepted ADR 상위 N개를
컨텍스트로 넣는다(일관성·중복 결정 방지).
3. `target_type`의 **템플릿**(P10 `templates/`)과 함께 LLM 호출.
4. 초안(마크다운, frontmatter 포함)을 받아 **lint를 미리 돌려** 결과와 함께 반환(저장은 안 함).

**출력**

```json
{
  "draft_markdown": "---\n...frontmatter...\n---\n## 배경 ...",
  "lint": {"ok": false, "reasons": ["대안 섹션이 비어 있음"]},   // 저장 전 참고용
  "used_sources": ["adr-0003", "upload:...", "..."],
  "related_context": ["adr-0002", "adr-0005"]                    // 근거로 참고한 기존 ADR
}
```

- **저장 안 함.** 반환된 초안을 UI 에디터에 띄우고, lint reasons를 표시(기획안 1 §9.2).
- 사용자가 편집 후 `PUT /docs/{id}` / `save_document`로 실제 저장(그때 정식 게이트·delta·이력).

## 3. 프롬프트 구조 (v0 — 튜닝 시작점)

**system (역할·규칙)**

- 너는 팀의 설계 결정을 기록하는 문서 작성자다. 목표 타입의 **템플릿 구조를 정확히** 따른다.
- **필수 섹션을 모두 채운다**: (ADR) 배경 / 결정 / 근거 / 대안 / 결과.
- **대안과 폐기된 선택지, 트레이드오프를 반드시 명시**한다. 이 프로젝트의 핵심은 "왜"다.
- **소스에 없는 사실을 지어내지 않는다.** 불확실하면 `> [확인 필요]`로 표시한다.
- 출력은 **유효한 frontmatter + 마크다운 본문만**. 설명·인사말 금지.

**user (조립 입력)**

```
[목표 타입] adr
[템플릿]
<templates/adr.md 내용>
[관련 기존 ADR — 일관성 참고]
<related_context 문서들의 결정/근거 요약>
[소스 자료]
<sources 취합 텍스트>
[지시] 인증 방식 결정을 ADR로. 위 소스에 근거해 초안을 작성.
```

> 위 문구는 v0다. 온도(temperature)는 낮게, 출력 형식 준수를 우선한다.
> 

## 4. 출력 계약 (하드 규칙)

- 반환 초안은 **frontmatter + 마크다운 본문**만(코드펜스·설명 없이).
- frontmatter 필수 필드 포함(`type`, `title`, `status: proposed`, `created` 등). `id`는 서버가 채번 규칙으로
임시 배정하거나 비워두고 저장 시 확정(택1 — 저장 시 확정 권장).
- 서버는 반환 전 **lint를 미리 실행**해 `lint.ok/reasons`를 함께 준다(막지는 않음 — 초안이므로).
- **저장은 하지 않는다.**

## 5. 가드레일

- **사람 검토가 게이트.** 생성물은 검토·수정 없이 저장될 수 없다(UI가 강제).
- **지어낸 근거 방지** — 소스 밖 주장은 `[확인 필요]`. 저장 전 사람이 해소.
- **LLM 미구성 시** `/generate`는 비활성(503/기능 숨김). 직접 작성(경로 C)은 항상 가능.
- 벡터 없음(관련 ADR 수집도 유사 RAG로).

## 6. 반복 튜닝 루프 (품질 확보)

- **골든 셋:** (소스 → 기대 초안) 몇 쌍을 만들어, 프롬프트 수정 시 초안 품질(필수 섹션 충족률, 지어냄 여부,
형식 준수)을 사람 평가로 추적.
- **프롬프트 버전 관리:** system/user 템플릿을 파일로 두고 버전 태깅. 변경 시 골든 셋 재평가.
- lint 통과율(초안이 곧바로 lint를 통과하는 비율)을 지표로 삼되, **최종 판단은 사람**.

## 7. 비범위 (v0에서 안 함)

- 자동 저장(항상 사람 검토 경유)
- 벡터/임베딩(유사 RAG로 관련 ADR 수집)
- 실제 프롬프트 문구 확정(튜닝하며 확정 — 여기선 v0 골격만)
- 멀티턴 대화형 생성(초기엔 단발 생성 → 사용자가 에디터에서 다듬기)
