# 구현스펙 — 멀티턴 AI 생성 (펑션콜/도구 호출)

## 1. 목적·배경

기존 M8 `generate_draft`는 **단발 호출**(폼 → 초안 1개 → 편집 → 저장)이다. 사용자가 AI와
**대화하며** 문서를 다듬고, 마지막에 "적용할까요?" → 확정 시 **추가/수정/삭제(=폐기)** 로 이어지게 한다.

대화 중 AI가 기존 지식에 근거하려면 검색·조회를 직접 수행해야 하므로 **펑션콜(OpenAI tools)** 방식을 쓴다.
읽기/컨텍스트 도구는 대화 중 **실시간 실행**, 변경(add/edit/deprecate)은 **staged changeset**으로만 캡처하고
실제 반영은 사람이 "적용"할 때 **승인 큐 제출**(구현스펙-승인워크플로우.md) → 관리자 승인 → `save`.

- §2-1 위반 없음: 대화·제안은 지식 store에 쓰지 않는다. 실제 쓰기는 승인 후 `store.save_document`만.
- §2-2: 읽기 도구는 기존 유사 RAG(`search_knowledge`, FTS5) 재사용. 벡터 없음.
- §5: 도구 구현은 전부 `KnowledgeService` 기존 메서드로 위임. 신규 검색/쓰기 로직 없음.

## 2. LLM 클라이언트 확장 — `hub/llm.py`

```python
def chat(self, messages: list[dict], *, tools: list[dict] | None = None) -> dict:
    """멀티턴 chat. tools 전달 시 function-calling. assistant 메시지(dict) 반환.
    반환: {"role":"assistant", "content": str|None, "tool_calls": [ {id, name, arguments(dict)} ] }"""
```

- 기존 `complete()` 유지(curate·generate_draft가 계속 사용).
- `available` 가드 동일. 미구성 시 `LLMUnavailable`(엔드포인트 503 매핑).

## 3. 도구 목록 (function schema)

### 읽기/컨텍스트 도구 — 실시간 실행, 결과를 tool 메시지로 대화에 append
| 도구 | 위임 | 용도 |
|---|---|---|
| `search_knowledge(query, k)` | `service.search_knowledge` | 관련 문서 탐색 |
| `get_document(id)` | `service.get_document` | 문서 원문·앵커 |
| `get_history(id)` | `service.get_history` | 과거 결정·근거 근거화(수정 시) |
| `get_related(id)` | `service.get_related` | 계보(supersedes/related) |
| `list_documents(type, status)` | `service.list_documents` | 목록·중복 확인 |
| `get_template(target_type)` | `_read_template` | 필수 섹션 구조 확보 |
| `lint_check(markdown)` | `_prelint` | 제안 전 자가 교정(§2-5 대안/폐기 강제) |

### 제안 도구 — 실행 안 함, staged changeset에 적재
| 도구 | 의미 |
|---|---|
| `propose_create(target_type, markdown)` | 새 문서 초안 스테이징 |
| `propose_edit(doc_id, markdown)` | 기존 문서 수정본 스테이징 |
| `propose_deprecate(doc_id, reason, superseded_by=None)` | 대상 문서 로드 → status=deprecated(+옵션 supersedes) markdown 스테이징 |

## 4. 대화 상태·세션

- 세션 store: **프로세스 인메모리** dict `session_id -> {messages[], staged[], actor, target_type}`.
  대화는 휘발성 워크플로우 상태로 지식 store/delta와 무관. **MVP 단일 프로세스 전제**(다중 워커·재시작 비영속 — 스펙 명시).
- `messages`: OpenAI 메시지 배열(system·user·assistant·tool). `staged`: 제안 도구가 쌓는 changeset.
- system 프롬프트: `generate_draft`의 규칙(필수 섹션·대안/폐기·근거·환각 금지) 재사용 + "도구로 근거를 모은 뒤 제안하라" 지침.

## 5. 오케스트레이터 — `hub/chat.py`

- 도구 루프: `llm.chat(messages, tools)` → `tool_calls` 있으면
  - 읽기/컨텍스트 도구 → 실행, `{role:"tool", tool_call_id, content}` append 후 재호출.
  - 제안 도구 → `staged`에 적재, tool 메시지로 "스테이징됨" 확인 append.
  - 반복 상한(예: 6) 초과 시 안전 종료.
  - `tool_calls` 없고 `content`만 있으면 → 사용자에게 반환하고 턴 종료.
- 안전장치: 제안 도구는 절대 store에 쓰지 않음. `lint_check`는 읽기 전용.

## 6. service API

- `chat_turn(session_id, user_message) -> {reply, staged}` — 없으면 세션 생성. LLM 미구성 시 `LLMUnavailable`.
- `get_session(session_id) -> dict | None`
- `apply_session(session_id, *, actor) -> [submission,...]` — `staged` 각 항목을 `submit_change(op,...)`로
  승인 큐 제출. 제출 후 세션 `staged` 비움.

## 7. 인터페이스

- **UI**: `/ui/chat`(화면), `POST /ui/chat`(HTMX 턴), `POST /ui/chat/apply`(staged→제출→"승인 대기").
- **HTTP**(옵션): `POST /chat`(session_id, message), `POST /chat/apply`.
- LLM 미구성 시 503 + "직접 작성/생성 폼 이용" 안내.

## 8. 테스트

- 가짜 LLM(tool_calls 방출) 주입 → 읽기 도구 실행 확인, 제안 도구 staged 적재(반영 안 됨).
- `lint_check` 자가 교정: 첫 초안 lint 실패 → 재제안 통과 흐름.
- `apply_session` → 승인 큐에 제출됨(pending). 승인 전 검색/조회 미노출.
