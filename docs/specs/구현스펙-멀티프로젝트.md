# 구현스펙 — 멀티프로젝트 (단일 DB + project 컬럼 스코프)

## 1. 목적·배경

지식 허브를 여러 프로젝트에서 쓰고 프로젝트별로 분리 관리한다. 인덱스에 `project` 컬럼이 예약만 돼
있었고(값 NULL) 배선되지 않았다(기획안2 사안). 한 인스턴스 안에서 **project 속성으로 논리 분리**하고,
검색·목록·채팅·생성을 현재 프로젝트로 스코프한다. 격리는 기존 필터-선행(§2-6)으로 강제한다.

## 2. 결정 (확정)

- **전역 유일 ID + project=속성**: `adr-0001`은 허브 전체에서 하나. project는 frontmatter 속성이자
  인덱스 컬럼이자 필터. **delta 엔진(paths/history/snapshot/lock/journal)과 `_next_id`는 미변경**.
- **자유 생성 project + 전역 관리자만**: 아무 project 문자열 허용(사전 등록 불필요). 승인은 기존 전역
  `admins`만. 제출은 project를 담아 승인함에서 프로젝트별로 걸러 본다.
- **기본 프로젝트 = `why-hub`**: 미지정(frontmatter에 project 없음 / 인덱스 NULL)은 default_project로 취급.

## 3. project의 원천과 흐름

- **원천은 frontmatter의 `project` 필드**. 파일이 원천이므로 reconcile(파일→인덱스 재구성) 후에도 보존된다.
- **기본 프로젝트 coercion**: 인덱스에 넣을 때 `project = fm.get("project") or default_project`.
  → frontmatter에 project가 없으면 자동으로 default_project. **기본 프로젝트 문서는 frontmatter를 안 건드림**
  (기존 파일·기존 테스트 무영향).
- **비기본 project 명시 시에만 frontmatter에 주입**(파일에 기록 → reconcile 안전).
- `normalize._FM_ORDER`에 `project` 추가(멱등성 위해 결정론적 순서).

## 4. 마이그레이션 (기존 NULL → default)

- `open_index(root, *, default_project)`: 연결 직후 `UPDATE documents SET project=? WHERE project IS NULL`.
  멱등(다음 open 시 매칭 0건). → 기존 인덱스 행(adr-0001~0003 등)이 즉시 default_project로 조회 가능.
- 이후 인덱스에 NULL project가 없으므로 필터는 단순 `project = ?`로 충분(SQL NULL 특수처리 불필요).

## 5. 인덱스 계층 (index_fts.py)

- `reindex_doc`: `documents` upsert의 project 값을 하드코딩 NULL → `doc.project`, `ON CONFLICT ... SET`에
  `project=excluded.project` 추가. (tenant는 계속 NULL — 기획안2 org 축, 범위 밖.)
- `list_projects() -> list[str]`: `SELECT DISTINCT project ... ORDER BY project`(NULL 제외).
- 필터/검색은 기존 `_EQ_FILTERS`(이미 `project` 포함)·`_candidate_ids`·`search`·`list_documents` 그대로.

## 6. 인터페이스 project 전달 규약

- **읽기**(HTTP): `?project=` 쿼리. 생략 시 전체(all). MCP: `search_knowledge` filters·`list_documents` project 인자.
- **쓰기/생성/채팅**(HTTP): 요청 바디 `project` 필드. 생략 시 default_project.
- **web**: 현재 프로젝트를 쿠키(`__project`)로. 상단 셀렉터(datalist=list_projects + 자유입력, "전체" 포함).
- **by-id 조회**(get_document/history/related): 전역 유일이므로 스코프하지 않음(직접 링크 동작).

## 7. 승인·채팅과의 결합

- 제출(submission)에 `project` 필드 저장. `list_submissions(status, project=None)` 필터. 승인은 전역 admins.
- 채팅 세션에 `project` 보관 → 읽기 도구가 그 project로 필터 → 현재 프로젝트 안에서만 근거. staged·apply가 project 전달.

## 8. config / 엔트리포인트

- `Config.default_project: str = "default"`(코드 기본). `config.toml`/example에 `default_project = "why-hub"`.
- main() 3종(web/http_api/mcp): `KNOWLEDGE_HUB_CONFIG` env 또는 `./config.toml` 있으면 `Config.load(...)`로 로드
  (멀티프로젝트·승인 설정이 실제 반영되게).

## 9. 테스트

- 인덱스: project 기록·필터·list_projects·open NULL→default 보정.
- 격리: 프로젝트 A 문서가 B 검색/목록에 안 뜸. 기본 프로젝트 coercion.
- ingest 프로젝트별 멱등, generate 관련검색 스코프, 채팅 세션 스코프, 제출 project 필터.
- 회귀: 기본 프로젝트 쓰기는 frontmatter 무변경 → 기존 스위트 그대로 통과.
